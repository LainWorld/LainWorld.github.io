Q：什么是负载均衡

A：负载均衡（Load Balancing）是一种在计算机网络或服务器系统中分配工作负载（即请求或流量）的方法。它的目的是将工作负载均匀地分配到多个服务器或计算资源上，以提高系统的性能、可靠性和可扩展性。

在高负载环境下，单个服务器可能无法满足所有请求或流量。负载均衡通过将请求分发到多个服务器上来解决这个问题。负载均衡器（Load Balancer）作为中间层设备，接收来自客户端的请求，并根据预定义的算法将请求转发到一个或多个后端服务器上进行处理。

负载均衡可以提供以下好处：

  1. 提高性能和可用性： 通过将工作负载分散到多个服务器上，负载均衡可以减轻单个服务器的压力，提高整体系统的响应速度和性能。如果某个服务器发生故障或不可用，负载均衡器可以将流量自动转发到其他正常运行的服务器上，从而提高系统的可用性和容错能力。
  2. 实现扩展性： 负载均衡允许根据需求动态地添加或删除服务器，以适应不断增长的流量或请求。通过水平扩展服务器数量，可以更好地处理高负载情况，提供更好的性能和吞吐量。
  3. 优化资源利用： 负载均衡器可以根据服务器的负载情况智能地分配请求，使每个服务器的负载保持相对均衡。这样可以最大程度地利用服务器资源，提高资源利用率，避免某些服务器过载而导致资源浪费。

常见的负载均衡算法包括轮询（Round Robin）、最少连接（Least Connection）、最短响应时间（Shortest Response Time）等。此外，负载均衡器通常还提供健康检查、会话保持、SSL终止等功能，以增加系统的可靠性和安全性。

负载均衡在许多领域得到广泛应用，包括Web服务器集群、应用程序服务器、数据库集群、内容分发网络（CDN）等，以提供高性能和可扩展的服务。

Q：什么是模块化，为什么要模块化

A：模块化是指将一个系统或软件分解为独立的、可组合的模块或组件的过程。每个模块都具有清晰定义的接口和功能，并且可以独立地开发、测试和维护。这种将系统划分为模块的方法可以带来许多好处。

模块化的原因：

  1. 可重用性： 模块化使得开发人员能够编写可以在不同项目中重复使用的模块。这样可以节省开发时间和资源，并提高软件的质量和一致性。
  2. 可维护性： 将软件划分为模块可以使得对特定功能或组件的修改和维护更加容易。当出现问题时，可以更精确地定位并修复特定模块，而无需修改整个系统。
  3. 可测试性： 模块化使得单元测试和集成测试更加容易。每个模块都可以独立地进行测试，从而更容易发现和修复错误。
  4. 并行开发： 在模块化的系统中，不同的开发团队可以同时开发不同的模块，而无需彼此过多地依赖。这样可以加快开发速度，并提高团队的协作效率。
  5. 降低复杂性： 将系统分解为模块可以降低整体系统

Q2：衡量模块独立的定性标准是什么？用自己的话表达其含义

A：衡量模块独立性的定性标准主要包括以下几个方面：

  1. 高内聚性（High Cohesion）： 模块内部的元素（例如函数、类）之间相关性越高，模块的内聚性就越高。高内聚性意味着一个模块的功能紧密相关，模块内的元素共享相似的目标和责任。模块内部高内聚性的好处是可以降低模块内部的复杂性，使得模块更易于理解、测试和维护。

  2. 低耦合性（Low Coupling）： 模块之间的依赖关系越少，模块的耦合性就越低。低耦合性表示模块之间的交互和依赖较少，模块可以独立地进行开发、测试和修改。低耦合性的好处是减少了模块之间的相互影响，提高了系统的灵活性、可扩展性和可维护性。

  3. 单一责任原则（Single Responsibility Principle）： 每个模块应该有一个清晰明确的责任或功能。一个模块应该专注于实现特定的任务或目标，而不是承担过多的责任。这样可以确保模块的功能明确，易于理解和修改。

  4. 抽象接口（Abstract Interface）： 模块应该通过抽象接口来定义与外部世界的交互。抽象接口将模块的实现细节与外部模块或组件解耦，使得模块可以独立地进行开发和测试。通过抽象接口，模块的实现可以被替换或扩展，而不会对其他模块产生影响。

    我的总结：高内聚则低耦合

Q3：如何理解信息隐藏和局部化？用自己的话或者例子表达其含义

A：

信息隐藏（Information Hiding）是一种设计原则，旨在将模块的实现细节和内部信息封装起来，只向外界暴露必要的接口和信息。通过信息隐藏，模块的实现细节对于外部模块或组件来说是不可见的，只有通过公开的接口才能与模块进行交互。这样做的目的是减少模块之间的耦合度，提高模块的独立性和可维护性。例如，一个类可以将其数据成员声明为私有的，通过公共的方法来访问和修改数据，这样可以隐藏内部数据的具体实现细节。

局部化（Localization）指的是将相关的功能或责任聚集在一起的过程。在局部化的设计中，一个模块或组件负责特定的功能或任务，并且尽量减少与其他模块之间的直接依赖关系。这样可以提高模块的内聚性，使得每个模块的功能清晰明确，易于理解和维护。例如，在一个电子商务网站的设计中，可以将用户管理、订单管理和库存管理等功能划分为独立的模块，每个模块负责相应的功能，彼此之间尽可能地减少直接的交互，从而实现局部化。

etc. 好比一个等级严森的军队，上级考虑的事情没有必要完全告诉下面的士兵（信息隐藏），只需要将部分指令（数据）发下给士兵，而上级里的敏感信息就不需要告诉下面的士兵。而应该把身体机能相近的士兵分在同一个队伍（密切相关的元素应该尽可能接近），当有任务执行时士兵收到上级的命令就开始分配指令给各个队伍，以队伍内规定的暗号来发配任务。

Q：为什么软件开发周期总是预估的2-3倍

A：因为软件开发过程中总是会遇到各种突发情况，各种问题会随着进度的推进而不断冒出。

  1. 软件开发的思维模式不能像数学一样简单的计算，软件开发过程没有公式可以套，不是按照一定的步骤就能得到想要的结果。
  2. 开发环境与过程会极大的影响进度。
  3. 需求理解有差异，用户表达的和程序员理解的不一样，出现偏差。
  4. 项目负责人任务分配的时间不合理。 有的任务分配时间过长，浪费了时间，有的任务分配时间过短，不够时间去完成。
  5. 预估开发周期的时候没有考虑到一些特殊情况，比如遇到法定节假日放假等。
  6. 也有可能团队中出现短板或者团队配合度不够

Q2：：什么是分而治之

A： 把一个复杂的问题分成多个相同或相似的子问题，再把子问题分成更小的子问题直到最后子问题可以简单地直接求解，原问题的解即子问题的解的合并。把复杂的问题简单化。

分治策略：对于一个规模为n的问题，若该问题可以容易的解决(比如规模n较小)则直接解决，否则将其分解为k个规模较小的子问题，这些子问题互相独立且与原问题形式相同，递归地解决这些子问题，然后将各个子问题的解合并得到原问题的解。

如果原问题可以分割成k个子问题，1<k<=n，且这些子问题均可解并且利用这些子问题的解求出原问题的解，那么分治方法就是可行的。由分治法产生的子问题往往是原问题的较小模式，这就为使用递归技术提供了方便。反复应用分治手段，可以使子问题与原问题类型一致而其规模却不断缩小，最终使子问题缩小到很容易直接求出其解。这自然导致递归过程的产生。分治与递归经常同时应用在算法设计之中。

使用场景

  1. 计算机科学中的并行计算： 在计算机科学领域，分而治之被广泛应用于并行计算中。例如，在图像处理中，一个大型的图像可以被分解成多个小块，并在每个小块上进行并行处理，最后将结果组合起来以获得整个图像的结果。这种方法不仅可以加速图像处理的速度，还能减少处理过程中的错误率。

  2. 生产制造中的生产流程管理： 分而治之也被广泛应用于生产制造中的生产流程管理。例如，在汽车制造中，整个生产流程可以被分解成多个部分，每个部分负责不同的任务，例如焊接、喷涂、组装等。这样一来，工人们可以更好地专注于自己的任务，从而提高了生产效率和产品质量。

  3. 自然资源管理中的环保: 在自然资源管理中，分而治之 also 被广泛应用于环保方面。例如，在水资源管理中，一个大型的水文系统可以被分解成多个小区域，每个小区域可以独立进行监测和管理。这种方法不仅可以提高监测和管理的精度，还能有效地保护水资源，减轻环境压力。

Q3：了解WBS

A：

WBS是 Work Breakdown Structure的缩写，意为 工作分解结构。它是一种项目管理工具，用于将一个大型项目分解成一个个更小、更易管理的任务和子任务。WBS可以帮助团队将整个项目拆分为可控制的部分，从而更容易实现项目目标。

WBS通常采用层级结构来组织任务和子任务，每个层级表示不同的任务级别。最高层级通常代表整个项目，随后的层级则逐渐展开并细化各个任务和子任务。WBS图表通常以树形结构呈现，使得整个项目在视觉上更加清晰明了。

优点

  1. 明确项目范围：WBS能够帮助团队明确项目的范围，将整个复杂项目拆分为可管理的任务和子任务，从而更好地把握项目进度和完成情况。

  2. 便于任务分配：WBS通过将项目任务和子任务分解到不同的层级中，使得任务分配更加清晰明了，每个团队成员都知道自己需要完成的任务和目标。

  3. 提高效率：WBS能够帮助团队更好地规划项目计划，提高团队成员之间的协作效率，并有效地避免浪费时间和资源。

  4. 易于监督：WBS能够为项目经理提供清晰的项目进度和完成情况的概览，从而使得项目管理更加容易监督和跟踪。

与分而治之的区别

与“分而治之”类似，WBS也是一种将大型任务分解成小型任务以便于处理的方法。但是，WBS与“分而治之”的区别在于其使用范围和目的不同。具体来说：

  1. WBS通常用于项目管理中，而“分而治之”可以应用于各种类型的问题。

  2. WBS是一种专业的项目管理工具，它需要经验和技能才能正确实施和使用。而“分而治之”是一种普遍适用的思维方式，几乎任何人都可以使用。

  3. WBS强调对工作进行结构化分解，以便更好地控制项目进展，避免漏洞。而“分而治之”则更注重在一定程度上将问题分解并分配给不同的人或小组，以便快速高效地解决它们。

Q：软件工程是不是教会不怎么会写程序的人开发软件？你的观点？

A： 我认为软件工程并不是教会不怎么会写程序的人开发软件的课程。

软件工程的目标是通过系统的学习，让我们了解软件开发的整个过程，包括问题定义、可行性研究、需求分析、总体设计、详细设计、编码和单元测试、综合测试及运行维护等阶段。软件工程更倾向于教会我们如何用规范的管理方法去管理自己的开发过程，而不是仅仅教授如何编写软件代码。

编程能力有所欠缺的人可以通过专门性的编程语言课程来提升，而不是从另一学科中去学习。因此，我认为软件工程并不是一门教会不怎么会写程序的人开发软件的课程
